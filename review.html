Executive Summary

The codebase has a functional microservice shape, but several trust boundaries are weak. The most serious problems are publicly writable product and stock endpoints, client-controlled order/payment/shipping state, tamperable prices and totals, and frontend XSS/CSRF risks around the chatbot.

Backend And API Findings

CriticalB-01 Product and category write APIs are public

Product service global REST permissions are empty, and create/update/delete handlers do not define a permission class.

Evidence: services/product-service/product_service/settings.py:88-94, services/product-service/products/views.py:22-30, :40-50, :64-69, :80-91.

Impact: anyone who can reach the product service or API gateway can create, modify, or delete catalog data.

Fix: use authenticated admin permissions for product/category writes. Keep only read endpoints public.

CriticalB-02 Stock reserve and release endpoints are unauthenticated

The product stock check/reserve/release views explicitly set empty authentication and permission classes, even though InternalServiceTokenPermission exists.

Evidence: services/product-service/products/views.py:94-118, :144-162, api-gateway/conf.d/default.conf proxies public /products/.

Impact: external callers can reserve stock to create false scarcity or release stock to inflate inventory.

Fix: require X-Service-Token or equivalent internal authentication on reserve/release; expose check as read-only if needed.

CriticalB-03 Order totals and item prices are trusted from client input

The order serializer accepts total_amount and per-item unit_price. The order service validates stock but does not fetch authoritative prices from the product service before saving.

Evidence: services/order-service/orders/serializers.py:6-37, services/order-service/orders/views.py:33-46, services/order-service/orders/services.py:89-99.

Impact: a direct API caller can create orders with manipulated totals or item prices.

Fix: accept only product IDs and quantities from clients. Fetch product prices server-side, calculate subtotal/shipping/total in the order service, and ignore submitted price fields.

HighB-04 Users can mutate orders after creation

The order detail endpoint checks ownership but lets the owner update all writable serializer fields, including status, total_amount, and shipping_address. It also allows direct delete.

Evidence: services/order-service/orders/views.py:57-69, services/order-service/orders/serializers.py:23-37.

Impact: users can cancel, alter, or corrupt order records outside the intended checkout flow.

Fix: expose narrow action endpoints for allowed transitions, or make user-facing order updates read-only after creation.

HighB-05 Payment records are client-controlled

Payment creation validates order access but accepts amount, provider, transaction_id, and status. The update endpoint repeats the same broad serializer behavior.

Evidence: services/payment-service/payments/serializers.py:6-10, services/payment-service/payments/views.py:51-58, :69-76.

Impact: a user can create or modify payment records that do not match the real order amount or payment processor state.

Fix: derive amount from the order service, make status processor-controlled, and make transaction IDs server/provider generated.

HighB-06 Shipment records are client-controlled

Shipment creation and update allow caller-supplied carrier, tracking number, and status after only checking order ownership.

Evidence: services/shipping-service/shipments/views.py:51-58, :69-76, services/shipping-service/shipments/serializers.py:6-10.

Impact: users can mark their shipment as pending/shipped/delivered or inject arbitrary tracking information.

Fix: restrict customer input to address fields; let shipping workflows or staff-only endpoints control carrier, tracking, and status.

MediumB-07 Product sorting requested by frontend is ignored by backend

The frontend sends ordering, but ProductService.list_products never reads or applies it.

Evidence: services/frontend_service/pages/views.py:141-149, services/product-service/products/services.py:47-84.

Impact: users select sorting options but results remain in database default order.

Fix: whitelist allowed ordering fields and apply queryset.order_by(...).

MediumB-08 Price filters use mismatched query names

The frontend sends price_min and price_max, while the product service expects min_price and max_price.

Evidence: services/frontend_service/pages/views.py:154-157, services/product-service/products/services.py:52-79.

Impact: product price filtering appears available in the UI but is ineffective.

Fix: align query parameter names or accept both names in the product service.

Frontend, Security, And Reliability Findings

CriticalF-01 Chatbot rendering enables XSS

The chatbot appends user text and AI responses with innerHTML after only replacing newlines. This is not sanitization.

Evidence: services/frontend_service/templates/base/base.html:157-177.

Impact: malicious text returned by the AI service, or typed by a user, can execute script in the storefront page.

Fix: render text with textContent and build line breaks as DOM nodes, or use a vetted sanitizer with a strict allowlist.

HighF-02 Chatbot proxy is CSRF-exempt

The frontend JavaScript sends a CSRF token, but the Django view is decorated with @csrf_exempt.

Evidence: services/frontend_service/pages/views.py:50-66, services/frontend_service/templates/base/base.html:130-138.

Impact: other sites can trigger chatbot requests through the user's browser, creating abuse and rate-cost exposure.

Fix: remove @csrf_exempt, require CSRF, validate content type, and add rate limiting.

HighF-03 Frontend service calls have no timeout

The base API client uses requests without timeout on every HTTP method.

Evidence: services/frontend_service/pages/api_client.py:25-55.

Impact: one slow or unavailable service can hang a web request until worker exhaustion.

Fix: add connect/read timeouts, retry only safe operations, and show user-friendly degraded states.

MediumF-04 Permissive CORS with credentials

The frontend service enables all origins and credentials together.

Evidence: services/frontend_service/frontend_service/settings.py:144-146; AI service also uses wildcard CORS at services/ai-service/main.py:52-59.

Impact: browser access policy is overly broad and easy to misconfigure into credential leakage.

Fix: set explicit allowed origins per environment and disable credentials unless required.

MediumF-05 Unsafe production defaults

The frontend defaults to a hardcoded secret and DEBUG=True.

Evidence: services/frontend_service/frontend_service/settings.py:27-33.

Impact: accidental deployment without environment variables exposes debug behavior and predictable signing secrets.

Fix: fail startup when required secrets are missing and default debug to false.

MediumF-06 Local fallback service URLs are wrong

Frontend defaults map user service to port 8001 and product service to port 8002, but compose maps product to 8001 and user to 8002.

Evidence: services/frontend_service/frontend_service/settings.py:154-161, docker-compose.yml service ports.

Impact: running the frontend outside Docker without environment overrides will call the wrong services.

Fix: align defaults with compose or require explicit environment variables.

LowF-07 Tailwind CDN is used in the application shell

The base template loads Tailwind through the CDN instead of a built CSS bundle.

Evidence: services/frontend_service/templates/base/base.html:8-13.

Impact: production pages depend on an external script and lose CSS optimization/control.

Fix: compile Tailwind during build and serve a static CSS asset.

UI And UX Findings

MediumU-01 Pagination links are malformed

The backend returns full next/previous links, but the template places that full URL inside ?page=....

Evidence: services/product-service/product_service/api.py:10-21, services/frontend_service/pages/templates/pages/products/list.html:103-115.

Impact: "Trang sau" and "Trang truoc" can generate URLs such as ?page=http://....

Fix: pass page numbers to the template, or render backend-provided URLs directly after normalizing host and query parameters.

MediumU-02 Checkout fields and selected methods are ignored

The checkout form collects full name, phone, shipping method, and payment method, but the view only reads shipping address and always creates standard shipping/COD payment.

Evidence: services/frontend_service/pages/templates/pages/checkout/checkout.html:20-99, services/frontend_service/pages/views.py:299-350, services/frontend_service/pages/api_client.py:213-244.

Impact: customers can choose express shipping or bank transfer, but the submitted order does not reflect their selection.

Fix: persist contact fields and honor selected shipping/payment methods in order, payment, and shipment creation.

MediumU-03 Cart badge always shows zero

The navbar hardcodes a zero cart count and the JavaScript update function is a TODO.

Evidence: services/frontend_service/templates/base/navbar.html:37-44, services/frontend_service/static/js/main.js:21-35.

Impact: users receive incorrect cart feedback after adding products.

Fix: expose cart count in the session context or fetch it with an authenticated endpoint.

LowU-04 Message alert classes do not map to Tailwind defaults

Django message tags like success and error are interpolated into Tailwind color class names, but Tailwind does not define colors named success/error by default.

Evidence: services/frontend_service/templates/base/base.html:21-29.

Impact: alerts may render without intended colors.

Fix: map message tags to real Tailwind colors such as green, red, yellow, and blue.

LowU-05 Mobile navigation lacks a compact search/menu flow

The search bar is hidden below the medium breakpoint and there is no mobile replacement. The account menu depends on hover, which is unreliable on touch devices.

Evidence: services/frontend_service/templates/base/navbar.html:12-29, :46-64.

Impact: mobile users lose primary search access and may struggle to open account links.

Fix: add a mobile search control and click/tap-driven account menu state.

Deployment And Configuration Findings

HighS-01 Hardcoded Neo4j credentials in compose

The Docker Compose file includes a static Neo4j password and repeats it in the healthcheck and AI service environment.

Evidence: docker-compose.yml:347-380.

Impact: shared credentials are easy to leak and hard to rotate across environments.

Fix: move credentials to environment variables or secrets, and fail startup when defaults are used outside development.
